## A. Intended Functionality
The Vault Knowledge Repository acts as the append-only, durable memory bank for the organization. It is responsible for ingesting, validating, and permanently storing approved organizational knowledge as referenceable artifacts for agent reasoning.

### Explicit Constraints
- **Append-Only Immutability:** Once a piece of knowledge is fully approved and committed, it cannot be edited. It can only be superseded by a newer version (with full lineage tracking) or soft-deleted via deprecation flags.
- **Source of Truth:** It holds no operational telemetry; it only holds verified strategies, SOPs, historical decisions, and curated intelligence.

## C. The Triple-Engine Extraction Logic
This feature entirely relies on the following engine mechanics:

### 1. OpenCLAW (Validation & Logic)
- **Trust & Novelty Evaluation:** Before insertion, evaluates if the proposed knowledge is actually novel compared to the existing Vector corpus, or if it contradicts existing trusted knowledge.
- **Lineage Verification:** Ensures the knowledge artifact has a valid cryptographic signature proving its origin (e.g., generated by Maya, approved by Human).

### 2. Goose (Extraction & Sequencing)
- **Extraction:** Parses raw operational chat logs, task outputs, or web extracts into structured, normalized `KnowledgeEntry` JSON formats (title, summary, vector_payload, citations).
- **Sequencing:** `Parse_Raw_Text -> Generate_Summaries -> Map_Taxonomy -> Present_For_Approval`.

### 3. NemoClaw (UI Automation & Navigation)
- **Anchoring:** Binds to the Repository Index grid, Content Diff panels (for superseding documents), and Approval/Reject CTAs.
- **Navigation Contract:** Enforces a split-pane review. The left pane shows the current existing knowledge, the right pane shows the proposed Goose-extracted knowledge.

## D. Conditional Navigation & Flows
1. **Ingestion Trigger:** An agent (e.g., Maya) determines a recent conversation holds strategic value.
2. **Extraction:** Goose compiles the conversation into a `KnowledgeEntry` format.
3. **Evaluation:** OpenCLAW cross-references the Vault context. If the data is 95% similar to an existing doc, it forces a "Supersede or Discard" UI prompt.
4. **Approval Gate:** The knowledge is placed in an SQLite purgatory tier. `nora` requests human validation in the internal Operations Chat.
5. **Commitment:** Once approved, the document is sealed into the Vault path with permanent lineage metadata.

## E. Validation, Error, and Exception Handling
- **Duplication Rejection:** If a document is a 100% duplicate, it is silently dropped and the UI toast indicates "Knowledge already exists."
- **Retrieval Threshold:** If a search query cannot find pertinent knowledge above an `0.85%` cosine similarity confidence, it returns an explicit `KnowledgeNotFound` exception, triggering an escalation to human operators rather than hallucinating.
- **Audit:** Cronjobs run weekly stale-knowledge audits, flagging docs unreferenced in 90 days for manual deprecation review.

## F. Hybrid Data Lifecycle
### SQLite (High-Performance/Ephemeral)
- Hosts the Vector database projection (e.g., pgvector/sqlite-vss) for ultra-fast local semantic retrieval.
- Caches the "Top 50 Most Accessed" knowledge objects for zero-latency prompt injection.
- Holds unapproved candidate knowledge pending the human-in-the-loop gate.

### Vault (Secure Commit State/Durable)
- The definitive Git-backed storage tier.
- Commits the Markdown/JSON knowledge objects, complete with cryptographic signatures preventing silent tampering.

# Phase 01 Plan 02: Decouple Virtual-Drive Policy Ownership

## Goal
Refactor Prana virtual-drive integration so runtime mechanism stays in core and policy ownership becomes client-managed (D-04 through D-08), with migration-safe behavior.

Purpose: Let Chakra/Dhi and future host apps control drive policy without forking core services.
Output: Policy contract wiring, startup/diagnostic integration, tests, and updated storage docs.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host policy config -> drive runtime core | Untrusted/invalid host policy can weaken mount or diagnostic guarantees. |
| Drive diagnostics -> startup gate | Severity mapping errors can produce unsafe non-blocking behavior or false blocking. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | E | drive policy injection | mitigate | Validate and type-check policy input at config boundary; enforce safe defaults when policy is missing/invalid. |
| T-01-05 | T | Vaidyar severity mapping | mitigate | Add explicit tests for policy-on/policy-off severity behavior and blocked signal translation. |
| T-01-06 | D | startup storage stage | mitigate | Preserve default fail-closed behavior and add regression tests for existing strict-path boot sequence. |
</threat_model>
